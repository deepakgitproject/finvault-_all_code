================================================================================
                          1. TEST STRATEGY OVERVIEW
================================================================================

1.1 Testing Approach
---------------------
  - Unit Tests: Test individual components in isolation (entities, handlers)
  - Integration Tests: Test components working together (API endpoints, repositories)
  - Test-Driven Development: Tests written before implementation (future)
  - Behavior-Driven Development: Given-When-Then format (future)

1.2 Test Pyramid
-----------------
  Unit Tests (70%)        - Fast, isolated, mock dependencies
  Integration Tests (20%) - Medium speed, real database, mocked external services
  E2E Tests (10%)         - Slow, full system, real browser/API calls

1.3 Test Organization
----------------------
  - Domain.Tests: Entity validation, business rules, state transitions
  - Application.Tests: Command handlers, query handlers, validators
  - Infrastructure.Tests: Repository implementations, services
  - API.Tests: Controller endpoints, middleware, authentication

1.4 Test Naming Convention
----------------------------
  Format: {MethodUnderTest}_{ScenarioOrInput}_{ExpectedBehavior}
  Example: Create_WithValidParameters_ShouldCreateUser
           Create_WithNullEmail_ShouldThrowArgumentException

    - Runs dotnet restore
    - Runs dotnet build
    - Runs dotnet test
    - Uploads code coverage to codecov.io
    - Fails build if tests fail

12.4 Test Best Practices
---------------------------

  1. Arrange-Act-Assert pattern
  2. One assertion per test (when possible)
  3. Descriptive test names
  4. Mock external dependencies
  5. Use test data builders
  6. Keep tests independent
  7. Clean up test data
  8. Use FluentAssertions for readable assertions
  9. Tag tests with categories ([Category("Unit")])
  10. Review tests in code reviews
